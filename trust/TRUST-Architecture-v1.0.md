## 一、三方信托关系

```
┌──────────────────────────────────────────────────────┐
│                    DATA TRUST TRIANGLE                │
│                                                      │
│   SETTLOR                    TRUSTEE                  │
│   (Fitness Member)  ────→  (ZWISERFIT Foundation)    │
│   - DID holder              - Data custody            │
│   - MPC key shard           - License management      │
│   - Consent grantor         - Compliance assurance    │
│         │                         │                   │
│         │                         ↓                   │
│         └──────────→  BENEFICIARY                     │
│                       (Member + Authorized Licensees) │
│                       - Data access (member)          │
│                       - Revenue share (member)        │
│                       - Licensed use (buyers)         │
└──────────────────────────────────────────────────────┘
```

### 角色定义

| 角色 | 身份 | 权利 | 义务 |
|------|------|------|------|
| **委托人 (Settlor)** | 健身会员（DID 持有者） | 数据主权、授权/撤销权、收益分配权、审计请求权 | 提供真实行为数据（自动采集） |
| **受托人 (Trustee)** | ZWISERFIT Foundation（独立法人） | 数据保管权、许可管理权 | 忠诚义务、保密义务、合规义务、信息披露义务 |
| **受益人 (Beneficiary)** | 会员本人 + 授权数据买家 | 数据访问权（会员） + 许可使用权（买家） | 不超出授权范围使用 |

## 二、防火墙架构

### 四层数据隔离

```
LAYER 1: 数据采集层 (On-Premise)
├── 门店IoT传感器 → 本地边缘计算节点
├── 原始视频 → 实时处理 → 丢弃（不离开门店）
├── 人脸向量 → 本地匹配 → 丢弃（不离开门店）
└── 行为事件 → 脱敏 → 加密 → 上传
                ↓
LAYER 2: 加密传输层 (TLS 1.3 + MPC)
├── 端到端加密隧道
├── 传输中数据不可被中间人截获
└── 到达受托人保管库前维持加密状态
                ↓
LAYER 3: 受托人保管库 (Encrypted at Rest)
├── 行为数据加密存储（AES-256-GCM）
├── MPC 密钥分片管理（受托人持有一片，会员持有一片）
├── 解密需双片签名 → 单方无法独立访问
└── 所有访问日志哈希存证
                ↓
LAYER 4: 授权访问层 (Smart Contract Gate)
├── 买家提交许可请求 → PDL 参数映射
├── 会员 MPC 签名授权（或拒绝）
├── 临时解密窗口打开（限时+限范围）
└── 访问完成后窗口关闭 → 哈希存证
```

### 防火墙原则

1. **原始数据不出门店** — 视频、人脸、PII 全部本地处理即丢弃
2. **一人一密** — 每个会员的 MPC 密钥分片独立，无万能密钥
3. **按需解密** — 数据默认加密，仅在被授权访问时临时解密
4. **全程存证** — 每一步操作产生哈希，纳入公开注册表

## 三、DRM3 四原则对齐

本信托架构严格对齐 DRM3 数据主权框架：

| DRM3 原则 | ZWF 实现 | 验证机制 |
|-----------|---------|---------|
| **Sovereignty（主权）** | DID + MPC 双片签名。会员拥有自己数据的控制权。 | 每次授权需会员 MPC 签名，授权记录链上可查 |
| **Privacy（隐私）** | 四层防火墙 + 数据最小化。行为数据脱敏不含 PII。 | Stella 独立隐私审计 + 零隐私泄露记录 |
| **Fairness（公平）** | 收益自动分账。会员按贡献比例获得许可收益。 | 智能合约自动执行，链上可审计 |
| **Transparency（透明）** | 公开注册表 + 变更日志。所有许可和审计记录公开可查。 | GitHub 公开仓库 + 哈希注册表 |

## 四、合规框架

### 4.1 中国大陆合规

| 法规 | 要求 | ZWF 实现 |
|------|------|---------|
| 《数据安全法》第36条 | 数据本地化存储 | 受托人保管库物理部署于中国大陆境内 |
| 《个人信息保护法》第38条 | 跨境传输安全评估 | 仅脱敏行为事件可跨境，需会员单独同意 |
| 《健康医疗大数据标准》 | 健康数据特殊保护 | 行为数据≠医疗数据，但仍采用等效保护标准 |

### 4.2 香港 / 新加坡双轨

| 法规 | 要求 | ZWF 实现 |
|------|------|---------|
| 香港《个人资料（隐私）条例》 | 数据保护原则 | ZWISERFIT Foundation 注册于香港，受该条例约束 |
| 新加坡 PDPA | 数据保护 + 可携权 | 国际数据买家适用新加坡 PDPA 标准 |

### 4.3 合规自检清单（外部审查用）

面向监管机构和数据买家的自助核查表：

- [ ] 数据采集：原始视频/人脸是否确认不离开门店设备？
- [ ] 数据传输：是否使用 TLS 1.3 + 端到端加密？
- [ ] 数据存储：是否 AES-256-GCM 加密？密钥是否分片管理？
- [ ] 数据授权：每次访问是否需要会员 MPC 签名？
- [ ] 数据删除：会员是否有"被遗忘权"的技术实现路径？
- [ ] 审计日志：所有操作是否有哈希存证？注册表是否公开？
- [ ] 破产保护：信托契约是否约束"数据不纳入破产财产"？
- [ ] 跨境传输：是否经安全评估？会员是否单独同意？

## 六、破产保护条款

本架构的核心保障——23andMe 死于"破产=卖数据"，ZWF 从结构上杜绝此风险：

1. **数据信托是独立法人** — 与运营公司资产隔离
2. **信托契约约束** — 受托人变更须经数据主体委员会同意
3. **继任受托人预指定** — 防止信托管理真空
4. **数据不作为破产财产** — 写入信托契约的不可撤销条款
